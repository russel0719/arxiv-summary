# 주간 논문 → 프로젝트 적용 매칭 — 작업 컨텍스트

이 지침은 그 주의 arXiv 일일 다이제스트(이미 선별·요약·태깅된 논문)와
`project_profiles.md`(내 프로젝트들의 task·스택·관심 태그·**개선 여지**)를 교차 분석해,
**"이 논문의 X 기법을 프로젝트 Y의 Z(개선 여지)에 적용하면 좋겠다"** 형태의
실행 가능한 주간 제안 리포트를 만드는 규칙이다.

## 입력

1. **이번 주 일일 다이제스트** — 셸이 프롬프트로 넘겨주는 `reports/YYYY/MM/report_DD.md` 파일들.
   각 논문 항목에는 제목·arxiv url·한줄요약·핵심기여·실무관련성·태그가 있다.
2. **프로젝트 프로파일** — `project_profiles.md`. 각 프로젝트마다 목적/Task/스택/관심 태그/개선 여지가 있다.

## 매칭 규칙

1. 이번 주 다이제스트의 **모든 논문**을 훑는다.
2. 각 논문을 프로젝트 프로파일과 대조한다. 매칭 신호(강→약):
   - **개선 여지 직접 대응** — 논문 기법이 특정 프로젝트의 "개선 여지" 항목을 직접 해결/개선 (가장 강함)
   - **태그 교집합** — 논문 태그 ∩ 프로젝트 관심 태그
   - **스택·task 인접성** — 같은 모델 계열/task라 전이·대체 가능
3. **억지 매칭 금지.** 관련 없으면 넣지 않는다. 한 논문이 여러 프로젝트에 걸치면 각각에 제안한다.
   한 논문이 어디에도 안 맞으면 조용히 건너뛴다.
4. abstract만으로 적용성이 애매한 **핵심 후보 1~2편**은 WebFetch로 프로젝트 페이지/본문을 확인해도 좋다(선택).
5. 제안은 **구체적**이어야 한다. "성능이 좋아질 수 있다" 같은 막연한 말 금지 —
   *어느 모듈/단계에, 어떤 방식으로, 무엇을 대체/추가*하는지 적는다.

## 우선순위 판정

- **높음**: 개선 여지의 핵심 병목을 직접 겨냥 + 통합 난이도 낮음/중간.
- **중간**: 관련은 명확하나 부차적이거나 통합 비용이 큼.
- **낮음**: 참고할 만하나 당장 적용 이득 불확실.

## 출력 형식 (고정 — 매주 동일 포맷)

셸이 지정한 출력 경로(`~/workspace/.meta/paper-match/weekly_match_YYYY-Www.md`)에 아래 형식으로 저장한다.

```
# 주간 논문 → 프로젝트 적용 제안 — YYYY-Www (KST)

- **스캔한 다이제스트**: N개 리포트 / 논문 M편
- **매칭 제안**: K건 (프로젝트 P개)

## 이번 주 하이라이트

<임팩트 큰 제안 2~3개를 불릿으로. "논문 → 프로젝트: 한 줄 이유">

---

## <프로젝트 경로> (예: products/matching/train-mambaglue)

### [논문 제목](arxiv url) — 우선순위: 높음/중간/낮음
- **적용 지점**: <프로파일의 어느 개선 여지에 대응하는지>
- **적용 아이디어**: <논문의 어떤 기법을 어느 모듈/단계에 어떻게 적용>
- **기대 효과**: <무엇이 얼마나 나아지는지, 정량 가능하면 정량>
- **태그**: <kebab-case 3~5개>

(프로젝트별로 매칭된 논문을 반복. 매칭 사이 `---` 구분.
 이번 주 매칭이 없는 프로젝트는 목록 끝에 "**이번 주 해당 없음**: projA, projB ..." 한 줄로 묶는다.)
```

- 프로젝트는 **매칭 건수 많은 순**으로 배치.
- 각 프로젝트 안에서는 **우선순위 높은 순**.
- 이번 주 유효 매칭이 0건이면, 헤더 + "이번 주 적용할 만한 신규 논문 없음" 한 줄만 남긴다.

(언어·git 등 공통 규칙은 repo 루트 `CLAUDE.md` 참조.)
